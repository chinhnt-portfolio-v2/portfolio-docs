# Epic 3 Retrospective - Live Evidence Layer

**Date:** 2026-03-14
**Status:** DONE

## Goal
Recruiter sees real-time health metrics on project cards — proving demo apps are actually running in production — with graceful degradation when data is offline.

## Completed Stories

| Story | Status | Notes |
|-------|--------|-------|
| 3-1: Health Metrics Polling Pipeline (BE) | ✅ | Polls health endpoints every 60s |
| 3-2: GitHub Webhook Metrics Refresh (BE) | ✅ | Triggers immediate re-poll on push |
| 3-3: WebSocket Server Metrics Broadcast (BE) | ✅ | Pushes updates to connected clients |
| 3-4: WebSocket Client Metrics Cards (FE) | ✅ | Real-time status updates on UI |
| 3-5: Metrics Staleness Auto-Hide (FE) | ✅ | Hides metrics after 5min inactivity |
| 3-6: Live GitHub Contribution Graph (FE+BE) | ✅ | Shows contribution count |
| 3-7: Config-Only Demo App Registration (BE) | ✅ | YAML-based app config |

## Key Learnings

### What Worked Well
1. **WebSocket broadcast** - Simple architecture, reliable real-time updates
2. **Health polling** - 60s interval provides good balance
3. **Demo apps config** - YAML-based approach is easy to maintain

### Challenges Encountered
1. **Duplicate beans** - DemoAppRegistry registered twice (@Component + @EnableConfigurationProperties)
2. **Jackson serialization** - Missing JavaTimeModule for Instant fields
3. **Cloud Run WebSocket** - Required proper endpoint configuration
4. **Vercel environment variables** - Needed explicit VITE_WS_URL for production

### Fixes Applied
- Removed duplicate bean registration in AppConfig
- Added JavaTimeModule to ObjectMapper in MetricsWebSocketHandler
- Added Vite proxy config for local development
- Configured Vercel environment variables for production
